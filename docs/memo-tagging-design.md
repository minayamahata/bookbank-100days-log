## 0. 目的と背景

### 0.1 何を解決するか

N0スパイク（`docs/n0-spike-results.md`）で確認したとおり、メモ本文からの**自動**キーワード抽出には構造的な限界がある:

- 日本語・韓国語は `NLTagger` の品詞タグ非対応で、助詞・形式名詞の混入をストップワード辞書とIDFで事後抑制するしかない
- 「私的な語」（例: 旅行の行き先、貸した相手の名前）が偶然一致すると、ユーザーの意図しない弱いエッジができる
- 逆に、ユーザーが「この2冊はつながっている」と思っていても、共通語がなければエッジはできない

**タグは、この自動抽出の裏返しとして「ユーザーが意図して本をつなげる」明示的な手段**を提供する。`#京都` と2冊のメモに書けば、その2冊は確実に・強く・理由つきでつながる。自動抽出が「機械の推測（弱い・玉石混交）」、タグが「人間の宣言（強い・確実）」という役割分担になる。

### 0.2 スコープの判断（Markdownではなくタグに絞る）

オーナーの要望は「タグや**Markdown的な要素**」だったが、初期スコープは **`#タグ` 記法のみ** とする **(仮)**。理由:

- 見出し・太字などの装飾記法は「本をつなげる」という目的に寄与せず、レンダラー実装（5言語・ダークモード・エクスポート互換）のコストだけが増える
- 既存のMarkdownエクスポート機能（`Utils/MarkdownExporter.swift`）との記法衝突を避ける検討が別途必要になる
- タグ単体なら正規表現1本で完結し、R4.5の小リリース規模に収まる

装飾記法が将来必要になれば、タグと独立に追加できる（本書の設計はそれを妨げない）。

## 1. 前提条件（確定・変更不可）

本書のすべての設計は以下の2条件を満たすこと。満たせない案は却下する。

| # | 前提条件 | 意味するところ |
|---|---------|--------------|
| 1 | **タグはメモ文字列内の記法であり、スキーマ変更ゼロ** | `UserBook.memo`（`String?`）・`MonthlyMemo.text`（`String`）は現状のまま。タグ用のSwiftDataモデル・プロパティ・リレーションを一切追加しない。タグはパース（表示時・インデックス生成時）で導出する |
| 2 | **R6（クラウド移行）と干渉しない** | 移行対象はプレーン文字列のままなので、移行ウィザード・Firestoreスキーマ（移行設計書 3.3節）・件数検証に変更は不要。タグを含むメモも含まないメモも、移行にとっては単なる文字列。R4.5がR6より先に出ても後に出ても成立する |

この2条件により、タグ機能は**ロールバック可能**（機能を消してもデータは無傷・`#京都` という文字列がメモに残るだけ）であり、**リリース順序の自由度を奪わない**。

## 3. 記法設計（T1）

### 3.1 記法の選定

| 案 | 内容 | 判定 |
|----|------|------|
| **A: `#タグ`** | Twitter/Instagram等で広く浸透。入力が軽い（1文字＋語）。5言語すべてで同じ記法 | **採用 (仮)** |
| B: `[[タイトル]]` | Obsidian式の直接リンク。本のタイトルを正確に入力する負担が大きく、表記ゆれで壊れる | 初期スコープ外（将来、タグで不足したら再検討） |
| C: `@メンション` | 「人」の含意が強く、本をつなげる用途に不自然 | 却下 |

「誰でも簡単に」という要望に対し、`#` は説明不要の記法として最有力。SNS経験のあるユーザーは教わらなくても使える。

### 3.2 パース仕様

```
タグ = "#" + 文字列（1〜30文字・正規化後）
終端 = 空白・改行・句読点（、。！？,.!?）・記号・別の "#"・文字列末尾
無効 = 数字のみ（#1, #2026）・空（# 単体）
```

- **正規表現ベース**で実装する。`NLTagger` 等のNLPは使わない → 日本語・韓国語の品詞タグ非対応問題（N0結果メモ・ノード設計書 3.1節）の影響を**受けない**。5言語で同一の挙動
- 抽出後の正規化はノードと同じ **NFKC＋小文字化**（`＃京都`＝`#京都`、`#SF`＝`#sf` を同一タグとする）
- パーサは**純関数**（`String → [Tag]`、Tagは表示位置レンジ＋正規化キー）としてユニットテスト対象にする（実装ガイド 5章）
- 既存メモに偶然含まれる `#` の誤検出は「数字のみ無効」＋「記号終端」でほぼ抑制できる（例: `#1位` は「1位」が数字始まりでも文字を含むため有効になる点に注意 — 許容する **(仮)**）

### 3.3 なぜスキーマ変更ゼロで成立するか

タグは「メモの中に既にある文字列」であり、データとして二重管理しない:

- **表示時**: メモ文字列をパースしてハイライト（`AttributedString`）
- **集計時**（サジェスト・絞り込み）: 全書籍のメモをパースしてタグ→本のインデックスをメモリ上に構築（425冊規模なら数ms・キャッシュはメモリのみ）
- **T2**: ノードのインデックス生成パイプラインが元々メモ本文を読んでいる（ノード設計書 3章）ので、その中でタグも抽出する — 保存先は既存計画の `bookIndex.keywords` であり、SwiftData/Firestoreのユーザーデータスキーマには触れない

## 4. 機能スコープ

### 4.1 T1（R4.5・v1.6.x）— タグ単体で価値が立つ最小セット

| 項目 | 内容 |
|------|------|
| パーサ | 3.2節の純関数＋ユニットテスト（5言語・境界ケース） |
| メモ表示 | 書籍詳細（`UserBookDetailView`）・メモ一覧でタグをハイライト表示（テーマ色・タップ可能） |
| 入力補助 | メモエディタ（`MemoEditorView`・`EditBookView`）で `#` 入力時に既存タグをサジェスト（表記ゆれの発生自体を抑える最大の対策） |
| タグ絞り込み | タグをタップ → 同じタグを持つ本の一覧（本棚のフィルター行 or シート。UIは実装時に確定 **(仮)**） |
| 対象 | `UserBook.memo`・`MonthlyMemo.text` の双方 **(仮)**（月メモのタグはその月に読んだ本のグルーピングとして機能する） |

**T1はノードグラフが存在しなくても単体で価値が立つ**（タグで本棚を横断整理できる）。これがR4.5として先行リリースできる根拠。

### 4.2 T1でやらないこと

- タグの編集・リネーム・削除の専用UI（メモ本文を編集すれば変わる。それがスキーマレスの利点）
- タグ一覧画面・タグクラウド（絞り込み導線で十分。利用状況を見て判断）
- Markdown装飾（0.2節）・`[[直接リンク]]`（3.1節）
- ノードグラフとの連携（T2 = R5）

## 5. ノードグラフ連携（T2・R5ノードβに同梱）

### 5.1 統合点

ノード設計書のパイプライン（正規化→言語判定→抽出→ストップワード→IDF）に対し、タグは**別レーンで合流**する:

```
メモ本文 ─┬─ 自動抽出レーン: NLTagger/フォールバック → ストップワード → IDF → keywords (source: "memo")
          └─ タグレーン:     正規表現パース（3.2節）→ そのまま採用   → keywords (source: "tag")
```

- タグレーンには**ストップワードもIDF減衰も適用しない**。ユーザーが意図して書いた語を機械が却下しない（前提7）
- `bookIndex.keywords` の要素に `source: "tag" | "memo" | "title"` を持たせる（R5は最初からFirestore保存＝N2統合のため、ローカルキャッシュの移行は発生しない。schemaVersionの初版に含める）

### 5.2 エッジ重み（順序のみ確定・2026-07-19）

**確定しているのは順序関係のみ**: シリーズ 1.0 > **タグ共有** > 著者 0.6。**タグの具体値はノードβの実データで調整する**（オーナー確定・2026-07-19）。

| 軸 | 重み | 備考 |
|----|------|------|
| シリーズ | 1.0 | 既存（ノード設計書 5章） |
| **タグ共有** | **1.0と0.6の間（具体値はノードβで調整）** | **新設**。「ユーザーの宣言」はシリーズ判定に次ぐ確実さ。reasons表示は「#京都」 |
| 著者 | 0.6 | 既存 |
| 共有キーワード | 0.15×IDF（3個上限） | 既存・自動抽出 |
| メモ同月 | 0.05 | 既存 |

- タグ共有はエッジ上限（k=8）・スコア下限の枠組みには従う（タグだけ特別扱いして毛玉化したら本末転倒）
- 同一タグを大量の本に付けた場合（例: 30冊に `#積読`）は完全グラフ化して毛玉になるため、**1タグあたりのエッジ生成上限 (仮: 共有本数が15冊を超えるタグはエッジ生成から除外し、絞り込み機能でのみ有効)** を設ける — 自動抽出のIDF足切りと同じ思想の、タグ版の安全弁
- タグ重みの具体値の調整には N0ハーネスを再利用できる（タグ軸を足して再走）

### 5.3 自動抽出との関係

タグが普及すれば「メモ由来の自動抽出語」の役割は縮小しうるが、**当面は併存** (仮・前提8)。タグを1つも使わないユーザーにも自動のつながりは価値があるため。ノードβでの実測（タグ利用率・雑音エッジ報告）を見て、メモ自動抽出語の重み引き下げ・廃止を判断する。

## 6. R6との非干渉の確認（前提条件2の検証）

| R6の構成要素 | タグの影響 |
|-------------|-----------|
| Firestoreスキーマ（`users/{uid}` 配下） | **なし**。`memo` / `text` はプレーン文字列のまま格納 |
| 移行ウィザード（WriteBatch・件数検証） | **なし**。タグ入りメモも単なる文字列として移行される |
| セキュリティルール | **なし**。タグ用のコレクションは存在しない |
| エンタイトルメント同期 | **なし**。タグは無料機能（前提6・確定） |
| Web v1（R8） | 閲覧側はWebでも同じ正規表現でハイライト可能（実装は任意・Web v1スコープ外でも文字列として普通に表示される） |

唯一の接点は T2 の `bookIndex.keywords.source` だが、これは R5（ノードβ）のスキーマであり R6 の移行対象ではない。

## 7. リスクと対策

| リスク | 対策 |
|--------|------|
| タグの表記ゆれ（`#京都` と `#京都旅行` が別タグ） | NFKC＋小文字正規化で機械的ゆれは吸収。意味的ゆれは入力サジェスト（4.1節）で発生自体を抑える。名寄せ機能は作らない（スキーマレスの範囲を超える） |
| 1タグの乱用による毛玉化 | 1タグあたりのエッジ生成上限（5.2節）。タグ数自体の上限は設けない **(仮)** |
| 既存メモの `#` 誤検出 | 数字のみ無効・記号終端（3.2節）。リリース前に実本棚データ（N0エクスポートJSON）でパーサを走らせ誤検出を目視確認する |
| Markdownエクスポートとの衝突 | `#` は行頭でMarkdown見出しになるが、エクスポートはメモを引用・本文として出力しており実害なし。エクスポート時にタグを装飾するかは実装時判断 **(仮)** |
| 5言語での入力性 | `#` は全キーボードで入力可能。ハングル・中文のタグはパーサが文字クラスで対応（3.2節）。テストデータはN0の5言語テスト棚を流用 |
